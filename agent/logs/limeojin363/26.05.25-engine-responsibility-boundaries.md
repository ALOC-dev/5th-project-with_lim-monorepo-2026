# 추천 엔진 책임 경계 재정리 후보

## 배경

최근 변경으로 `RecommendationEngine`은 더 이상 `unknown` 입력을 직접 스키마 검증하지 않고,
사용처가 `UserInputSchema.parse(...)`로 검증한 `UserInput`을 넘긴다고 취급하게 되었다.
또한 엔진은 외부 logger를 주입받지 않고 `loggingActivated` boolean만 받아 내부에서
`consoleLogger` 또는 `noopLogger`를 선택한다.

이와 같은 방향으로 엔진 전체를 훑었을 때, 더 나아질 수 있는 책임 경계 후보를 정리한다.

## 현재 관찰

- `RecommendationEngine`은 전체 retry orchestration, 초기 discovery query 생성, step 호출,
  error output 변환, secrets 분배, 로거 선택을 함께 담당한다.
  - 대상: `packages/recommendation-engine/src/v1/engine.ts`
- `discoverSeeds`는 현재 `_userInput`을 인자로 받지만 실제로 사용하지 않는다.
  실제 입력은 `DiscoveryContext`와 provider options다.
  - 대상: `packages/recommendation-engine/src/v1/steps/discoverSeeds/index.ts`
- `evaluateSeeds`는 후보 평가 orchestration 외에도 live enrichment client 생성,
  scrape cache 구성, Playwright browser lifecycle, reference URL resolution,
  LLM scoring 호출, 로그 payload 상세 구성까지 같이 담당한다.
  - 대상: `packages/recommendation-engine/src/v1/steps/evaluateSeeds/index.ts`
- 일부 내부 함수가 schema parse를 수행한다. public boundary 검증인지 내부 invariant 검증인지
  의미가 섞일 수 있다.
  - 예: `DiscoveryContextSchema.parse`, `DiscoverSeedsOutputSchema.parse`,
    `EvaluateSeedsOutputSchema.parse`, `PlaceRecommendationItemSchema.parse`
- 실패 분류가 error message 문자열에 의존한다.
  - 대상: `packages/recommendation-engine/src/v1/steps/evaluateSeeds/utils/failure.ts`
- `mapWithConcurrency`가 여러 파일에 중복된다.
  - 대상: `evaluateSeeds/index.ts`, `evaluateSeeds/llm/scoring.ts`,
    `evaluateSeeds/llm/enrichment.ts`

## 개선 후보

### 1. `discoverSeeds`에서 `userInput` 제거

`discoverSeeds(_userInput, context, logger, options)`의 `_userInput`은 현재 사용되지 않는다.
입력 책임은 이미 engine의 discovery context 생성 단계에서 끝난다. 따라서 `discoverSeeds`는
`DiscoveryContext`와 provider dependency만 받도록 좁히는 편이 맞다.

제안:

- `discoverSeeds(context, logger, options)`로 시그니처 변경
- 엔진 호출부에서 `this.userInput` 전달 제거
- 테스트/사용처 타입체크로 영향 확인

효과:

- step의 실제 입력 계약이 명확해진다.
- unused `_userInput`이 암시하는 잘못된 책임을 제거한다.

우선순위: 높음

### 2. 엔진에서 초기 discovery query 생성을 분리

현재 `engine.process()`가 `createDiscoveryContextWithLlm(...)`를 직접 호출한다.
이는 orchestration과 LLM 기반 query planning 책임이 섞인 상태다.

제안:

- `planDiscoveryQueries(userInput, config, secrets)` 또는 `createInitialDiscoveryState(...)`
  같은 작은 모듈로 분리
- 엔진은 `EngineDiscoveryState`를 받고 retry loop를 수행하는 쪽에 집중
- 장기적으로는 discovery planner를 테스트 가능한 순수 boundary로 만들기

효과:

- 초기 query 생성 실패와 retry orchestration 실패를 분리하기 쉽다.
- LLM planner 교체나 fallback planner 추가가 쉬워진다.

우선순위: 높음

### 3. `evaluateSeeds`에서 live dependency assembly 분리

`evaluateSeeds`는 현재 평가 pipeline 함수이면서 동시에 live client/caches/browser를 만든다.
이 때문에 순수 평가 흐름과 외부 I/O 조립이 같은 파일에 묶여 있다.

제안:

- `createLiveEvaluateSeedsDependencies(options, logger)` 모듈을 별도 분리
- `evaluateSeeds`는 다음과 같은 dependency를 options로 받도록 정리
  - `enrichCandidates`
  - `resolveReferenceUrls`
  - `scoreCandidates`
- live 기본값은 상위 factory에서 조립

효과:

- 평가 pipeline의 단위 테스트가 쉬워진다.
- browser/cache/API credential lifecycle이 명확한 boundary로 이동한다.
- live/stub/test dependency 전환이 쉬워진다.

우선순위: 높음

### 4. `secrets` 전달을 step별 dependency로 대체

현재 engine이 secrets를 알고 step별로 필요한 필드만 다시 전달한다.
이 구조는 secrets shape가 엔진 전체에 퍼지기 쉽다.

제안:

- engine option은 `secrets` 대신 `dependencies` 또는 `providers`를 받는 방향 검토
- 최소 단계로는 `discoverSeeds`/`evaluateSeeds` options 내부에서 secrets를 직접 쓰는 부분을
  provider factory로 이동

효과:

- engine은 secret field 이름을 덜 알게 된다.
- 외부 API 교체와 테스트 stub 주입이 쉬워진다.

우선순위: 중간

### 5. 내부 schema parse의 의미를 분리

사용처 입력 검증은 boundary에서 수행하고, 엔진 내부는 검증된 타입을 취급하는 방향으로 바뀌었다.
같은 기준을 내부 단계에도 적용하면 schema parse의 의미를 분명히 해야 한다.

구분 제안:

- public/input boundary parse: 외부 입력을 타입으로 좁히는 책임
- internal invariant assertion: 엔진이 생성한 값이 contract를 만족하는지 개발 중 확인하는 책임
- output contract parse: 외부로 내보내기 직전 안정성 보장 책임

제안:

- 내부 생성 값 검증 함수는 이름을 `assert*Contract` 또는 `parse*OutputContract`처럼
  의도를 드러내기
- 단순 타입 보장을 위해 반복 parse하는 곳은 제거 가능 여부 검토

효과:

- "누가 검증 책임을 갖는가"가 코드 이름에 드러난다.
- schema parse 실패가 사용자 입력 문제인지 내부 버그인지 구분하기 쉬워진다.

우선순위: 중간

### 6. 실패 분류를 문자열 추론에서 typed error로 이동

`toEvaluateSeedsFailure`는 error message에 `"Invalid input"`, `"zod"`, `"parse"`가 있는지로
스키마 오류를 추정한다. 이는 책임 경계가 불명확하고 오탐 가능성이 있다.

제안:

- zod error는 `error instanceof ZodError`로 분류
- 내부 domain error는 `EngineStepError` 같은 typed error로 분류
- vendor/API/LLM/schema/postprocess 실패를 명시 타입으로 관리

효과:

- 실패 코드가 안정적이다.
- error message 변경에 덜 취약하다.

우선순위: 중간

### 7. 공통 concurrency helper 분리

`mapWithConcurrency`가 여러 파일에 중복된다. 현재 로직은 거의 동일하지만 normalize 방식이 조금씩 다르다.

제안:

- `src/v1/utils/async.ts` 또는 step-local shared util로 `mapWithConcurrency`를 분리
- concurrency normalization 정책을 하나로 고정

효과:

- 작은 중복 제거지만, 외부 I/O 안정성 정책을 한 곳에서 관리할 수 있다.

우선순위: 낮음

### 8. 로그 payload 구성과 도메인 흐름 분리

`evaluateSeeds` 내부의 enrichment success log는 분석용 상세 payload를 많이 만든다.
이 코드가 핵심 평가 흐름을 가린다.

제안:

- `toEnrichmentSuccessLog(...)`, `toReferenceRejectedLog(...)` 같은 serializer 유틸로 분리
- 도메인 흐름은 enrichment/ranking/output 생성만 읽히게 정리

효과:

- 평가 flow 가독성이 좋아진다.
- 로그 포맷 변경이 도메인 로직 변경처럼 보이지 않는다.

우선순위: 낮음

## 추천 진행 순서

1. `discoverSeeds`에서 unused `userInput` 제거
2. `evaluateSeeds` dependency assembly 분리
3. 초기 discovery query planner 분리
4. typed error / schema parse 의도 정리
5. concurrency/log serializer 같은 작은 정리

## 주의점

- 이번 방향은 "엔진 내부에서 아무 검증도 하지 않는다"가 아니라,
  "public input 검증은 사용처가 담당하고, 엔진 내부 검증은 내부 invariant/output contract 확인으로
  의미를 제한한다"로 보는 것이 적절하다.
- `RecommendationEngine`이 live 실행을 기본으로 조립하는 현재 구조를 갑자기 모두 DI로 바꾸면
  API가 커질 수 있다. 우선은 step 내부 factory 분리부터 시작하는 편이 안전하다.
- 기존 테스트 스크립트는 trace 수집을 `testMonitor`에 의존한다. `loggingActivated`만 남긴 현재
  정책과 test trace 수집 요구가 충돌할 수 있으므로, 추후에는 console log를 파싱하지 말고
  테스트 전용 runner가 별도 event sink를 가질지 결정해야 한다.
