# 클라/서버/전체 프로젝트 선제 정비 후보

## 배경

현재 레포는 pnpm workspace 기반의 TypeScript monorepo다.

- client: Vite + React + Emotion + React Router
- server: Express + tsx dev + esbuild build
- packages: `api-contracts`, `recommendation-engine`
- 현재 기본 스크립트: `dev`, `build`, `typecheck`, server 중심 `test`
- 아직 lint/format/test runner/CI/문서화/환경변수 표준화는 거의 비어 있음

4인 팀 기준으로, 팀원이 어렵지 않게 접근할 수 있어야 한다. 따라서 도구는 "강력함"보다
"설명 가능함, 실행 명령이 단순함, 실패 메시지가 이해 가능함, 점진 도입 가능함"을 우선한다.

## 추천 도입 원칙

- 처음부터 모든 것을 강제하지 말고 `format`, `lint`, `typecheck`, `test`, `build` 순서로 넓힌다.
- 로컬 개발 명령은 루트에서 대부분 실행 가능하게 둔다.
- CI는 로컬 명령과 동일한 명령을 호출하게 만든다.
- 복잡한 모노레포 오케스트레이터는 필요가 생길 때 도입한다.
- 팀원이 "왜 실패했는지" 바로 알 수 있게 README와 스크립트 이름을 단순하게 유지한다.
- 자동화는 권장 단계와 강제 단계를 나눈다. 초기에는 formatter만 강제해도 효과가 크다.

## 전체 프로젝트 후보

### 1. 포매터

- Prettier 도입
  - 효과: 코드 스타일 논쟁 제거
  - 난이도: 낮음
  - 추천 우선순위: 매우 높음
  - 후보 패키지: `prettier`
  - 스크립트:
    - `format`: 전체 포맷
    - `format:check`: CI용 포맷 검증
- `.prettierrc` 추가
  - 기본값을 너무 많이 커스터마이즈하지 않는 편이 좋음
  - 예: `semi`, `singleQuote`, `trailingComma`, `printWidth`
- `.prettierignore` 추가
  - `dist`, `coverage`, `node_modules`, generated file 제외
- import 정렬 플러그인 검토
  - 후보: `@ianvs/prettier-plugin-sort-imports`, `prettier-plugin-organize-imports`
  - 주의: import 순서가 팀 피로도를 만들 수 있으므로 초기에는 보류 가능

### 2. 린터

- ESLint flat config 도입
  - 효과: 버그성 코드, React hooks 오용, unused import, 접근성 문제 조기 발견
  - 난이도: 중간
  - 추천 우선순위: 높음
  - 후보 패키지:
    - `eslint`
    - `typescript-eslint`
    - `eslint-plugin-react-hooks`
    - `eslint-plugin-react-refresh`
    - `eslint-plugin-jsx-a11y`
    - `eslint-plugin-import-x` 또는 `eslint-plugin-simple-import-sort`
- 루트 `eslint.config.js`에서 workspace 전체를 다룬다.
- 처음부터 너무 엄격하게 하지 말고 warning과 error를 구분한다.
- 추천 error:
  - React hooks rules
  - `no-undef`
  - TypeScript parser 오류
  - unreachable / duplicate cases
- 추천 warning:
  - unused vars
  - import order
  - a11y 일부 항목
- 스크립트:
  - `lint`
  - `lint:fix`

### 3. 타입체크 표준화

- 루트 `typecheck`는 이미 존재하므로 유지.
- 각 package/app의 `typecheck` 스크립트 명칭을 통일한다.
- `tsconfig.base.json`의 공통 옵션을 명확히 관리한다.
- 점진적으로 `strict` 계열 옵션 확인
  - `strict`
  - `noUncheckedIndexedAccess`
  - `exactOptionalPropertyTypes`
  - `noImplicitOverride`
  - `noFallthroughCasesInSwitch`
- 주의: 한 번에 켜면 팀 피로도가 높으므로 package별로 단계 적용.

### 4. Git hook

- Husky + lint-staged 도입
  - 효과: 커밋 전 포맷 자동화
  - 난이도: 낮음~중간
  - 추천 우선순위: 높음
  - 후보 패키지: `husky`, `lint-staged`
- 초기 hook은 formatter만 추천
  - `prettier --write` on staged files
- lint/test를 pre-commit에 모두 걸면 느려서 팀원이 우회하기 쉽다.
- `pre-push`에 `typecheck` 정도는 선택 검토.

### 5. CI

- GitHub Actions 또는 사용 중인 Git 플랫폼 CI 도입
  - 효과: main branch 안정화
  - 난이도: 중간
  - 추천 우선순위: 높음
- 최소 job:

## 클라이언트 후보

### 1. 라우팅 구조 정리

- 현재 React Router 도입 상태.
- routes를 한 파일에 둘지, page별 route module로 분리할지 결정.
- 현재 규모에서는 `src/router.tsx` 단일 파일 유지 가능.
- 화면이 늘어나면 `src/routes/placeRecommendation.tsx` 같은 domain route file 검토.
- route path 상수화 검토
  - `ROUTES.placeRecommendation.form`
  - 링크 오타 방지.

### 2. 페이지/컴포넌트 폴더 규칙

- 현재 `src/page/RecommendationForm/page.tsx` 패턴이 있음.
- 규칙 제안:
  - `page/*/page.tsx`: route component
  - `features/*`: 기능 단위 state/hooks/components
  - `shared/ui`: 범용 UI
  - `shared/lib`: 작은 utility
  - `shared/api`: API client
- 팀이 작으므로 FSD 같은 큰 방법론을 그대로 도입하기보다 가벼운 domain-first 구조 추천.

### 3. 전역 스타일 초기화

- `modern-css-reset` 이미 도입됨.
- 추가 검토:
  - `html, body, #root { min-height: 100%; }`
  - 모바일 viewport 대응
  - safe-area inset 정책
  - font smoothing
  - 기본 font-family
- 이 부분은 reset 라이브러리와 별도의 `global.css` 또는 Emotion `Global`에서 관리 가능.

### 4. 디자인 토큰

- `tokens:generate` 스크립트가 이미 있음.
- Figma DS와 코드 토큰의 소스 오브 트루스 결정 필요.
- 후보:
  - CSS variables 생성
  - TS token object 생성
  - Emotion theme 생성
- 색상/typography/spacing/radius/shadow/z-index 계층화 추천.
- 토큰명은 Figma와 코드가 최대한 일치해야 함.

### 5. UI 컴포넌트 정책

- atomic design을 쓰기로 했던 흐름이 있으므로 코드에서도 최소 계층 정리 검토.
- 예:
  - atoms: Button, Icon, Text, Input
  - molecules: SearchInput, DateChipGroup
  - organisms: BottomSheet, PlaceList
- 단, page 종속 organism은 page 근처에 둔다.
- 공통화 기준:
  - 2회 이상 반복
  - variant가 명확함
  - page context 없이 설명 가능함

### 6. 접근성

- ESLint `jsx-a11y`
- semantic HTML 사용
- button/link 구분
- modal focus trap
- bottom sheet aria policy
- keyboard navigation
- color contrast check
- axe 테스트 검토
- 모바일 앱이어도 웹 접근성은 유지 필요.

### 8. E2E/브라우저 테스트

### 11. API client 계층

## 서버 후보

### 1. Express 앱 구조 분리

- 현재 `src/index.ts`가 entrypoint일 가능성이 높음.
- 추천 구조:
  - `app.ts`: express app 생성
  - `index.ts`: env parse + listen
  - `routes/*`: route handlers
  - `middleware/*`: cors/error/logging
- 테스트에서 `app`만 import하면 supertest가 쉬워짐.

### 2. 서버 lint

- 전체 ESLint에 server override 추가.
- Node globals, Express types, async handler 규칙.
- 추천 rule:
  - no-floating-promises 검토
  - consistent-type-imports
  - no-misused-promises는 Express handler에서 false positive 주의.

### 3. 서버 테스트

- Vitest 또는 Node test runner 선택.
- 4인 팀 접근성 기준으로 Vitest 추천.
- Supertest 도입 검토
  - route integration test
  - health endpoint
  - recommendation endpoint
- 테스트 종류:
  - unit: request mapper, validation, error mapping
  - integration: express route
  - contract: response schema
  - smoke: server start + health.

### 4. API contract

- `packages/api-contracts`가 있으므로 적극 활용.
- request/response schema를 zod로 정의.
- server는 inbound parse, outbound parse 또는 assertion.
- client는 response parse.
- API contract package가 client/server 사이 타입 source of truth가 되게 관리.

### 5. OpenAPI 문서

- zod schema에서 OpenAPI 생성 검토
  - 후보: `zod-openapi`, `@asteasolutions/zod-to-openapi`
- API가 외부 공유될 예정이면 가치 큼.
- 내부 팀만 쓰면 초반에는 endpoint markdown으로 충분.

### 6. Request validation

- middleware 또는 route-local parse 정책 결정.
- body/query/params 구분.
- validation error response shape 표준화.
- 400 응답의 field error 구조를 client가 쓰기 쉽게 정리.

### 7. Error handling

- 중앙 error middleware 추가.
- typed error 도입 검토:
  - `BadRequestError`
  - `NotFoundError`
  - `UpstreamError`
  - `InternalError`
- error response shape 통일:
  - `code`
  - `message`
  - `details`
  - `requestId`

### 8. Logging

- `pino` 또는 `winston` 검토.

### 9. Request ID

## 도입 시 주의할 점

- 지금 당장 가장 큰 구멍은 "스타일"보다 "일관된 검증 명령과 온보딩"이다.

- 테스트는 서버/엔진 pure logic부터 시작하고, 클라이언트는 route/page smoke와 form interaction부터 시작한다.
- Figma design system과 코드 token은 한 번에 완전 자동화하려 하지 말고, 이름/범위부터 맞춘다.
- server와 engine의 책임 경계는 이미 논의가 있었으므로 API validation과 engine input boundary를 문서화해야 한다.
- `dist`, `.env`, `.DS_Store` 같은 파일의 추적 여부는 별도 정리가 필요하다.
